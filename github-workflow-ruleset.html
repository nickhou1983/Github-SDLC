GitHub Actions Workflow 是什么

一个 .github/workflows/*.yml 文件即一条工作流：由 事件触发,拆成多个 作业（Job）,在 运行器（Runner） 上按 步骤（Step） 顺序执行。

# 层级结构
Workflow （工作流）
└─ Event 触发事件（on）
└─ Job 作业 · 默认并行
    └─ Runner 运行器（runs-on）
    └─ Step 步骤 · 顺序执行
        └─ uses: Action 或 run: 命令

因事件而动

push / pull_request / schedule / workflow_dispatch 等触发运行。

并行 + 依赖

多 Job 默认并行，用 needs 声明先后形成 DAG。

产出信号

Actions 运行产生 checks,被 Ruleset 用作合并门禁。

// 03 · RULESET07 / 14

什么是 Ruleset（规则集）

一套 命名的规则列表,控制人们如何与仓库中的 分支 / 标签 / 推送 交互。可在 Repo / Org / Enterprise 三层级配置（每仓库 / 每组织最多 75 个）。

Branch 分支

谁能 push、是否要 PR / 检查 / 签名、禁强推等。

Tag 标签

谁能创建 / 删除 / 重命名标签，命名模式约束。

Push 推送

按文件路径 / 大小 / 扩展名阻止推送（含 fork 网络）。

相较 Branch Protection 的优势

可叠加多个同时生效 · 有状态（active / evaluate / disabled）无需删除即可开关 · 任何读权限者可见 · 支持组织 / 企业级统一下发。

// 04 · CHECKS & DEPLOY10 / 14

状态检查与部署门禁

合并 / 更新到目标分支前，强制先通过以下四类验证之一或组合：

required_status_checks

必需检查全绿才可合并。strict=分支须先更新到最新；可用 integration_id 限定只信任某 App。

required_deployments

合并前须成功部署到指定环境（如 staging）,环境需预先在 Settings 定义。

workflows

锁定具体 workflow 文件(path + repository_id + ref/sha)必须通过，防同名检查绕过。

merge_queue

经合并队列排队、基于最新基分支验证后再合并。ALLGREEN / HEADGREEN 分组策略。

// 04 · CHECKS & DEPLOY11 / 14

状态检查到底检查什么

核心认知

Ruleset 本身不定义检查内容。它只强制"名为 X 的检查必须为 passing"；检查什么由产生它的工具决定（写在你的 workflow / CI 里）。

状态从哪来

Checks：GitHub App / Actions 产生，有行级注释；Commit statuses：外部 CI 经 status API 上报。

常见"内容"

构建 · 单测 / 集成测试 · Lint / 类型检查 · 覆盖率 · CodeQL 安全扫描 · 外部 CI · 自定义校验。

坑 1 · 名字不匹配

ruleset 的 context 必须与上报的检查名完全一致，否则永远"等待"。

坑 2 · skipped=Success

被跳过的任务报告为 Success，即使必需也不阻止合并。

// 05 · BYPASS13 / 14

GitHub App 与 Bypass 的能力边界

想做的事能否说明 / API
直接绕过规则（bypass actor）✓ 可以以 actor_type: Integration 加入 bypass list；PUT .../rulesets/{id}
审批 / 拒绝 bypass request✗ 不行审核人只能 team / role / user；无审批写入端点,仅 UI / 通知
列出 / 监控 bypass request✓ 只读GET /orgs/{org}/bypass-requests/push-rules

⚠ 仅安装不够

App 装到 Enterprise / Org 不会自动获得绕过权限，必须在对应 ruleset 的 bypass list 中显式添加。

自动放行的正解

不要让 App"审批别人"；应把 App 设为 bypass actor,由它自己执行操作绕过规则。

维度传统 ActionAgentic Workflow

计费仅 Actions 分钟数Actions 分钟数 + AI 推理（AIC）

选传统 Action

构建 / 测试 / Lint · 发布部署 · 格式检查 · 定时备份——步骤已知、要可复现与合规。

选 Agentic Workflow

Issue 分诊 · 调查 CI 失败 · 状态报告 · 文档同步 · 重构迁移——目标明确但需阅读理解与判断。

// SUMMARY14 / 14

关键 Takeaways

01 · AI 交付链路

Coding Agent 开 PR → Code Review 评审 → Agent Merge 合并；全程过 Ruleset 门禁。

02 · 分工

Workflow 负责自动执行并产出检查信号；Ruleset 负责设门槛约束交互。

03 · Ruleset 分层

Repo / Org / Enterprise 三层可叠加；enforcement 支持 evaluate 先观察再强制。

04 · 检查本质

Ruleset 只管"哪些名字必须绿"，真正检查逻辑在你的 workflow / CI 里。

05 · Bypass 边界

App 可作 bypass actor 直接绕过 与 只读监控,但不能审批 request。

06 · Agentic Workflow

自然语言 Markdown + AI 智能体跑在 Actions 上；safe-outputs + Ruleset 双层护栏。
